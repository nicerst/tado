# CloudFox

CLI that maps cloud attack surface (IAM permissions, exposed resources, privesc paths) across AWS, Azure, and GCP by querying live cloud APIs with the credentials already on hand.

## When to use / when NOT to use

- Use for enumeration/recon during an authorized cloud pentest or security review: inventorying IAM permissions, finding exposed resources, mapping cross-account/cross-role privesc paths.
- Do NOT use for exploitation — CloudFox explicitly does not automate exploitation (unlike Pacu). Recon/enumeration only.
- Do NOT use as a compliance or benchmark tool — no alerting, no baseline scoring (that's ScoutSuite/Prowler/Security Hub territory).
- Do NOT run against any account without explicit authorization to assess it — see Rules below, this is a hard gate, not a suggestion.

## Setup

Requires v1.17.0+ (older versions broken by an AWS format change).

```bash
brew install cloudfox
# or
go install github.com/BishopFox/cloudfox@latest
# or download from GitHub Releases
```

**Per-cloud prerequisites (must already be configured before invoking CloudFox):**

- **AWS**: AWS CLI configured (profile / env vars / EC2 instance metadata). Recommended IAM policy = `SecurityAudit` managed policy + bundled `misc/aws/cloudfox-policy.json` custom policy (SecurityAudit alone misses AppRunner, Grafana, Lambda GetFunctionURL, Lightsail). ReadOnlyAccess is broader than needed. If AdministratorAccess is what's actually granted, flag that as a finding in itself.
- **Azure**: existing Azure CLI session, Viewer-equivalent role.
- **GCP**: `gcloud auth application-default login`. Minimal = `roles/viewer` for single-project recon. Org-wide assessment needs: `organizationViewer`, `iam.securityReviewer`, `cloudasset.viewer`, `cloudidentity.groupsViewer` at org level, `folderViewer` at folder level. 10 GCP APIs must be individually enabled first (Cloud Identity, Cloud Asset, Resource Manager, IAM, Compute, Secret Manager, Cloud Functions, Cloud Run, GKE, BigQuery).

## Core usage

**Run everything (AWS quick start):**
```bash
cloudfox aws --profile [profile-name] all-checks
```

**Multi-account:**
```bash
cloudfox aws -l <file-of-profile-names> all-checks   # newline-separated profile list
cloudfox aws -a all-checks                            # all stored profiles
```

**Targeted module + filter:**
```bash
cloudfox aws access-keys --profile test_account
cloudfox aws access-keys --filter access_key_id --profile readonly_profile
```

**Cross-account privesc path analysis** (requires pmapper graph data already generated by a separate `pmapper` run — CloudFox consumes it, doesn't generate it):
```bash
cloudfox aws cape -l file_with_profile_names.txt --admin-only
```

**Opt-in noisy check** (KMS excluded by default — slow/noisy):
```bash
cloudfox aws resource-trusts --include-kms
```

**Azure pattern** — run `whoami` first to get tenant/subscription IDs, then feed into other modules:
```bash
cloudfox azure whoami
cloudfox azure inventory --subscription-id <id>
cloudfox azure rbac --subscription-id <id>
cloudfox azure storage --subscription-id <id>
```

**GCP**: use `roles/viewer` for single-project recon (`iam`, `permissions`, `instances`, `gke`, `cloudrun`, `functions`, `bigquery`, `cloudsql`, `storage`, `vpc-networks`, `firewall`, `lateral-movement`, `data-exfiltration`, `public-access`, `privesc`, `hidden-admins`). Escalate to the org-level role matrix only when the assessment scope is org-wide, not per-project.

**Global AWS flags**: `--profile/-p`, `--profiles-list/-l`, `--all-profiles/-a`, `--yes/-y` (non-interactive), `--output/-o` (brief|wide), `--verbosity/-v` (1-3), `--outdir` (default `~/.cloudfox`), `--max-goroutines/-g` (default 30), `--skip-admin-check`, `--cached/-c` (loads disk cache — faster but risks stale data), `--mfa-token`, `--pmapper-data-basepath`.

## Rules / gotchas

**Authorization gate (blocking, non-negotiable):** CloudFox queries live APIs against a real AWS/Azure/GCP account for virtually every command — it does not operate on offline/exported data (even the `cape` privesc-path command traces back to a prior live-credentialed pmapper run). It can only enumerate accounts you already hold valid credentials for, but **having valid credentials is not the same thing as having authorization to run a security assessment against that account.** Before running any CloudFox command, confirm explicit authorization to assess the target account — mirror vapt-init's Step 0 authorization gate: state the target account/tenant/project, confirm authorization status explicitly, and stop if authorization is absent or unclear. Do not proceed on the assumption that "I have creds for it" is sufficient.

- Silent failures in black-box/found-credential mode are **by design**: checks that fail do so silently, so any data returned means the credentials in use have the access needed to retrieve it. Absence of output is itself informative — don't treat it as a bug.
- Check `~/.cloudfox/cloudfox-error.log` for noisy errors — two documented causes: (1) unsupported region for a service (CloudFox probes every region regardless), or (2) insufficient IAM permissions / SCP block.
- Most commands print a table AND write a "loot file" to disk with pre-populated follow-up commands (e.g. `aws s3` commands for discovered buckets, curl commands with embedded tokens for API gateways, `aws eks update-kubeconfig` for discovered clusters). Check the loot file, not just stdout.
- `--cached` is faster but may miss recent changes (documented in the flag's own help text) — avoid it when findings need to reflect current state.
- Never treat a clean CloudFox run as a compliance pass — it has no alerting or baseline scoring; pair with a compliance tool if that's the actual requirement.
